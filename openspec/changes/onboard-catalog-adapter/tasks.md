## Reconciliation Decisions (resolving design.md's Open Questions — binding, not left to apply)

- [ ] R.1 **Both-given / neither-given error class**: reuse the existing `ManifestError` (no new exception type). Wording: both-given → `ManifestError("onboard accepts either a client name or --manifest, not both")`; neither-given → `ManifestError("onboard requires either a client name or --manifest <path>")`. Both are raised and caught by the same `except ManifestError as exc: typer.echo(f"error: {exc}", err=True)` block `onboard` already uses for every other failure — no new except clause needed for this case. Rationale: matches the single `error:` line convention already used verbatim by `validate`/`onboard`/`lock`/`project`/`unlock` in `manifest.py`, and both messages are pure user-input-shape errors, not I/O or resolution failures, so `ManifestError`'s existing semantics fit without inventing a subclass.
- [ ] R.2 **`CatalogSourceError` message class**: kept as its own exception type (`odoo_forge_catalog/errors.py`), NOT folded into the resolver's `invalid-catalog` failure type. Rationale: `invalid-catalog` is a `ProjectCatalogResolutionFailure` value returned by `ProjectCatalogResolver.resolve()` for a *matched-but-incomplete* record; a `CatalogSourceError` (missing/malformed `catalog.yaml`, malformed record schema) happens **inside** `find_matches()`, before the resolver ever sees a match — synthesizing a fake `invalid-catalog` result for it would require fabricating a `ProjectCatalogResolutionFailure` the resolver never produced, which the adapter must not do (spec: adapter "MUST NOT perform resolution... failure classification"). Instead, `onboard`'s catalog-driven branch adds one more `except CatalogSourceError as exc: typer.echo(f"error: {exc}", err=True)` clause, mirroring how `run`/`onboard` already stack sibling exception families (`ManifestError`, `CredentialError`, `RegistryError`, `BackendError`) each in their own catch, all rendering the identical single `error: {exc}` line. This keeps `CatalogSourceError` visibly distinguishable from `catalog-not-found`/`ambiguous-resolution`/`invalid-catalog` in output, per the design's original (undecided) leaning.
- [ ] R.3 **Positional `<cliente>` → `ProjectCatalogRequest` mapping**: the CLI builds `ProjectCatalogRequest(client_key=client)` only (no `project_key`/`project_slug`/`manifest_name` populated from the single positional arg). Rationale: matches the proposal's framing ("ask for a client by name alone") and the spec's scenario language ("resolves uniquely for client identifier `acme`" / "more than one catalog record matches the supplied client identifier") — both describe single-dimension (`client_key`) matching; multi-dimension disambiguation stays a `ProjectCatalogRequest` capability the CLI does not need to expose yet (no flags for it in scope).
- [ ] R.4 **Catalog-driven backend invocation defaults**: `instance="default"` and no digest override (`odoo_image=None`), matching `run`'s own defaults — `onboard <cliente>` does not currently expose `--instance`/`--odoo-image-ref` flags (out of scope; not in proposal/spec). Document this as a known follow-up if multi-instance catalog onboarding is ever needed.
- [x] R.5 **`CatalogIndex` Protocol runtime-checkability**: `odoo_forge/project_catalog/interfaces.py`'s `CatalogIndex` is a plain (non-`@runtime_checkable`) `Protocol`. Both the catalog-index-adapter spec's "Structural conformance via isinstance" scenario and the composition-factory spec's "Factory returns a protocol-conforming instance" scenario require `isinstance(x, CatalogIndex)` to succeed, which raises `TypeError` against a non-runtime-checkable `Protocol`. Add `@runtime_checkable` to `CatalogIndex` (mirrors `PublishedArtifactResolver`'s existing shape). This is a one-decorator, behavior-preserving addition to core — NOT one of the frozen symbols (`ProjectCatalogResolver`, `plan_backend`, `DockerBackendProvider.run`, `plan_projection`, `project_workspace`) the proposal says must not be modified, so it is in scope.
