# Proposal: Stabilize the Real-Docker Baseline

## Intent

Replace the skipped real-daemon test with lifecycle evidence before dependent stabilization work, without changing the canonical contract.

## Business and Operational Value

- Prove the stack starts, reports status, stops safely, and leaves no test-owned leaks.
- Detect regressions while keeping Docker opt-in, with optional separate CI.

## Scope

### In Scope
- Replace the integration-test skip with a disposable real-Docker `run -> status -> stop` provider scenario.
- Use an Odoo image generated by this project's factory and an official pinned PostgreSQL image.
- Prove `stop` preserves lifecycle volumes, then perform final test-owned cleanup that deletes only resources owned by the test.
- Record Docker versions, commands/results, and residual checks during verification.
- Consume the readiness-fix bootstrap implementation as the production prerequisite for the passing lifecycle receipt.

### Out of Scope and Non-Goals
- Production or contract changes, including lifecycle and volume semantics.
- A universal merge gate or Docker dependency for the default test suite.
- Custom PostgreSQL images, broad CLI coverage, or duplicated normative requirements.
- Fixing production defects discovered by the harness; such findings stop this unit and require a separate SDD change.

## Capabilities

### New Capabilities
None.

### Modified Capabilities
None. This change supplies runtime evidence for `openspec/specs/local-backend/spec.md` without changing its requirements.

## Approach

Build a uniquely identified, secret-safe provider fixture with ephemeral ports and bounded readiness. Skip only for unavailable Docker prerequisites; later failures fail. Use unconditional cleanup and independent residual checks, distinguishing preserved lifecycle volumes from test-owned deletion.

## Affected Areas

| Area | Impact | Description |
|---|---|---|
| `tests/adapters/test_docker_provider_integration.py` | Modified | Lifecycle harness and cleanup evidence |

## Dependencies

- Reachable Docker daemon; project-factory Odoo image; pinned official PostgreSQL image.
- Roadmap Unit 1 and the canonical `local-backend` spec.
- Acyclic order: implement `fix-odoo-factory-health-readiness` first; this baseline then supplies its acceptance evidence. The harness is not an implementation-completion dependency.

## Chained Delivery Boundary

Forced Feature Branch Chain with one test-only child work unit under 400 authored lines. The rollback boundary is the harness and test-owned fixtures; production files are excluded.

## Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Slow/unavailable images or daemon | Medium | Prerequisite skip and bounded cold-boot timeout |
| Resource or credential leakage | Medium | Unique ownership, secret-safe fixture, `finally` cleanup, residual checks |
| Harness exposes production defect | Medium | Stop and open a separate SDD change |

## Rollback Plan

Revert the harness and test-owned fixtures without touching production behavior or user resources.

## Acceptance Outcomes

- [x] Default suite remains unchanged and excludes integration tests.
- [x] Real-Docker test completes `run -> status -> stop` using the required images.
- [x] Stop preservation and ownership-scoped cleanup are proven, with no test-owned leaks or exposed credentials.
- [x] Verification records Docker versions and exact command results.
